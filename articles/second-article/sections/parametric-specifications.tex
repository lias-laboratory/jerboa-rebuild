During an object construction, a Parametric Specification records the rules applied and their parameters (both topological and geometric) in order to describe the modelling process. %
When a Parametric Specification is edited, rules may be added, deleted, moved, and their parameters can be changed.%
With such changes, the topological parameters may have different concrete names, be deleted and so on. %
As a result, using the parameters' concrete names eventually lead to unexpected results (at best) or failure at runtime. %

To this end, Persistent Names are required to robustly identify topological parameters across an object reevaluation. %
The following Parametric Specification:

\begin{verbatim}
    1 - square(pos);
    2 - extrude(3, vec);
    3 - insert(41, vec);
    4 - extrude(42, vec);
    5 - triangulate(8);
    6 - collapse(63);
    7 - chamfer(55);
    8 - colour(10);
\end{verbatim}

represents the modelling process of an initial evaluation
(Fig.~\ref{fig:evaluation}). %
Its reevaluation (Fig~\ref{fig:reevaluation}) proposes a variant modelling
process where a rule is added between applications 1 and 2, and where the
application 3 is deleted. %


\subsection{Persistent Name}

With rules using darts as topological parameters, it ensues that these are the
entities Persistent Names must robustly identify. %

A dart is created through a unique sequence of rules' applications and their
matching nodes. %
Thus, a Persistent Name being a record of $\PN{application, rule\_node}$ pairs
does robustly identify a dart. %

For example, following the initial evaluation, dart $10$'s
(Fig.~\ref{fig:7-eval}) Persistent Name is $\PN{1n_{3}-2n_{1}-5n_{0}}$. However,
due to an editing of the parametric specification, this same dart is given a
different concrete name $c$ (Fig.~\ref{fig:7-reeval}) at reevaluation. %
At evaluation, $square$'s (application 1) rule node $n_{3}$ creates the dart $3$
''ex nihilo``. %
Then, $extrusion$'s (application 2) rule node $n_{3}$ copy creates dart $10$
from dart $3$. %
And, finally, dart $10$ is matched by $triangulation$'s (application 5) rule
node $n_0$. %
At reevaluation, $squares$'s rule node $n_{3}$ recreates the dart $3$
identically. %
Now, the added $insertion$'s rule node $n_{0}$ matches the dart $3$ and, there
is a shift in the concrete names attributions from now on. %
Then, $extrusion$'s rule node $n_{3}$ copy creates the dart $12$ from the dart
$3$. %
And, finally, dart $12$ is matched by $extrusion$'s rule node $n_{0}$. %
Thus, the Persistent Names makes it possible to find that dart $c$ concrete name
is $12$. %

By construction, a Persistent Name identifies a unique dart. %
That is, darts all have a different Persistent Name. %

\subsection{Parametric Specification syntax}

A number of fields are used to describe an application within a parametric
specification, namely an application number \textit{AppNumber}, a \textit{Rule},
the topological arameters \textit{TopoParameters} and the geometric paramaeters
\textit{Geometric Parameters}. Additionnaly, a set of tags if an application was
present at evaluation (\textit{INIT}), or if it either added (\textit{ADD}),
deleted (\textit{DELETE}) or moved (\textit{MOVE}) at reevaluation. %

%\begin{verbatim}
%    [TAG, ApplicationNumber, Rule, TopologicalParameters, GeometricParameters]
%\end{verbatim}

Thus, the Parametric Specification of the initial modelling process
(Fig.~\ref{fig:evaluation}), once edited to produce the reevaluation process
(Fig.~\ref{fig:reevaluation}), takes the following form:

\begin{verbatim}
    INIT   1-square(pos);
    ADD     -insert(3);
    INIT   2-extrude(PN1={1n3}, vec);
    DELETE 3-insert(PN2={1n5;2n4}, vec);
    INIT   4-extrude(PN3={1n5;2n4;3n0}, vec);
    INIT   5-triangulate(PN4={1n3;2n1});
    INIT   6-collapse(PN5={1n4;2n5;4n6});
    INIT   7-chamfer(PN6={1n5;2n5;3n0;4n4;6n2});
    INIT   8-colour(PN7={1n3;2n3;5n0});
\end{verbatim}
