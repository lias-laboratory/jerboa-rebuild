\section{Evaluation}

Once the initial evaluation (Fig.~\ref{fig:evaluation}) is done and its
parametric specification \maxime{init-spec} built, an evaluation directed
acyclic graph (or DAG) must be issued for each registered persistent name before
a parametric specification can be re-evaluated. %
An evaluation DAG traces the history of each topological parameter back to the
first created orbits it originates from, thus allowing the matching of the
corresponding topological parameter at re-evaluation time. %


\subsection{Evaluation DAG}

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.3\textwidth]{pn3}
  \caption{$PN_{3}$'s evaluation DAG}
  \label{fig:eval-dag-pn3}
\end{figure}

Although a persistent name represents the history of a dart, an orbit is subject
to topological changes and, thus, requires the construction of its own history
in order to be accurately matched at re-evaluation. %
To this end, an evaluation DAG is built with the applications and nodes
registered in a persistent name from end to start. %

An evaluation DAG is arranged by layers representing the different applications
registered in a persistent name. %
Each layer is made of: An orbit level, indexed by a node and containing orbits;
An event level, indexed by a rule's application number and contains events. %

For example, $PN_{3}$'s evaluation DAG (Fig.~\ref{fig:eval-dag-pn3}) represents
the history of a face $\orbit{0,1}$. %
Considering $PN_3=[1n_{5};2n_{6};3n_{0}]$, its evaluation DAG starts with the
application $3$: The orbit level is indexed by $n_{0}$ and contains a face orbit
$\orbit{0,1}$; The event level above is indexed with the application number
$3$. %
A syntactic analysis of the extrusion rule for the face $\orbit{0,1}$-orbit
incident to $n_{0}$ reveals that it was \emph{modified} by the application $3$
so the contained event is a $MODIFICATION$. %
Continuing with application $2$: The orbit level is indexed with $n_{6}$ and
since the face orbit was modified (\textit{i.e.} no creation nor split nor
merging of the orbit), it can be traced up to this level; The event level, this
time, is indexed by $2$ and a query will find that the face orbit was created by
this application. %
Finally, for the application $1$: The orbit level is indexed with $n_{5}$ and
since in the application $2$ the face $\orbit{0,1}$-orbit incident to $n_{6}$
originates from another face, the current orbit level contains a
$\orbit{0,1}$-orbit; The event level above is indexed by $1$ and a syntactic
analysis of the square rule reveals that the face $\orbit{0,1}$-orbit was
created. %
The construction of the DAG finishes after application 1 since the left-hand
side of the rule does not contain any node. %

A bulletin board containing the syntactic analysis of a modeller's rules can be
queried in order to obtain events and automatically build an evaluation DAG. %


\subsection{Traces and origins}

In order to accurately represent the history of an orbit, two types of links are
used in an evaluation DAG (and later on in the re-evaluation DAGs): a trace link
(black) to link an orbit with its previous state and an origin link (red) to
link an orbit with its origin. %

An origin orbit is a sub-orbit common to both the traced orbit and its previous
state in the history which, through the application of a rule, leads to the
creation of the traced orbit. %
The type $\orbit{o'}$ of an origin orbit can be deduced through the syntactic
analysis of a rule. %
For example, in the triangulation rule (Fig.~\ref{fig:rule-triangulation}), the
face $\orbit{0,1,3}$-orbit incident to $n_{0}$ is split. %
In this case, the origin $\orbit{o'}$-orbit contains the $n_{0}$'s implicit
links relabelled into $0$, $1$ or $3$ in the preserved nodes of the
$\orbit{0,1,3}$-orbit incident to $n_{0}$ in the right-hand side of the rule. %
In the left-hand side, $n_{0}$ is labelled $\orbit{0,1,3}$. %
In the right-hand side, $n_{0}$ is the only preserved node and is labelled
$\orbit{0,\_,3}$ so $0$ and $3$ belong to $\orbit{o'}$. %
Hence, the origin orbit of a face split by the triangulation rule is a face edge
$\orbit{0,3}$. %

Therefore, an origin is an orbit which helps alleviate the ambiguity at
re-evaluation by designating the orbit that led to the generation of an orbit
(in the case of the creation and the split) or the merging of two orbits. %

When the orbit is either \emph{modified} or \emph{not modified}, it can be
straightforwardly traced up to the previous application despite the topological
change because the orbit type did not actually change. %
For example, in $PN_{3}$'s evaluation DAG (Fig.~\ref{fig:eval-dag-pn3}), the
face $\orbit{0,1}$-orbit is modified by the application $3$ which is the insert
vertex rule. %
Since there is no ambiguity about the orbit being traced, the face
$\orbit{0,1}$-orbit can be traced up straightforwardly to the previous
application. %
These two states of the same orbit are linked in the history with trace (black)
arrow. %

\begin{figure}[hbt]
  \centering
  \includegraphics[width=0.4\textwidth]{pn7}
  \caption{$PN_{7}$'s evaluation DAG}
  \label{fig:eval-dag-pn7}
\end{figure}

When the orbit is either \emph{created}, \emph{split} or \emph{merged} a
syntactic analysis also allows the deduction of an \emph{origin} in addition to
a traced orbit. %
For example, in the $PN_{7}$'s evaluation DAG (Fig.~\ref{fig:eval-dag-pn7}) the
face $\orbit{0,1}$-orbit incident to the application $2$'s node $n_{4}$ is split
by the application $5$ which is the triangulation rule
(Fig.~\ref{fig:rule-triangulation}). %
In this case because the face orbit is split into faces there will be an
incertitude at re-evaluation about which face to choose. %
The origin orbit, a face edge $\orbit{0}$-orbit, is found to be a necessary and
sufficient orbit to designate which face will need to be matched at
re-evaluation. %


\subsection{Paths to origins}

\begin{figure}[hbt]
  \centering
  \includegraphics[height=35pt]{rule-face-collapse}
  \caption{Face collapse rule}
  \label{fig:rule-face-collapse}
\end{figure}

In some cases, usually in a merging event, the origin orbit can be deleted after
the application of a rule and the origin does not represent the right orbit. %

For example, in the collapse rule (Fig.~\ref{fig:rule-face-collapse}), the
origin to the vertex $\orbit{1,2,3}$-orbit incident to $n_{2}$ in right-hand
side originates from the face $\orbit{0,1,3}$-orbit incident to $n_{0}$
(illustrated if Fig.~\ref{fig:5-eval} and Fig.~\ref{fig:6-eval}). %

Now, considering the $PN_{6}$, whose application $6$ is the collapse rule, the
origin to the vertex $\orbit{1,2,3}$-orbit, a face $\orbit{0,1,3}$-orbit, is
expected to be incident to the application $4$'s node $n_{4}$. %
However, the face incident to $n_{4}$ is adjacent to the face matched by $n_{0}$
(Fig.~\ref{fig:demo-fig-extrusion-right-match}) although the application is
supposed to collapse the top face matched by $n_{6}$ (Fig.~\ref{fig:6-eval}). %

\begin{figure}[hbt]
  \centering
  \includegraphics[width=0.4\textwidth]{pn6}
  \caption{$PN_{6}$'s evaluation DAG}
  \label{fig:eval-dag-pn6}
\end{figure}

In order to circumvent such a limitation, a \emph{path to the origin} must be
maintained in the history. %
For example, in $PN_{6}$'s evaluation DAG (Fig.~\ref{fig:eval-dag-pn6}), a path
$@1.@2.$ is associated to the face $\orbit{0,1}$-orbit incident to the
application $4$'s node $n_{4}$. %
Since an origin is always in the hook node, a path to the origin represent a
traversal from the current node, $n_{2}$ in this case, to the hook node. %
As a result, the traversed explicit links from $n_{2}$ to $n_{0}$ are, in the
following order, $1$ and $2$. %
In the Jerboa notation, this path between is written $@1.@2$. %
Concretely, this means that the origin orbit to the vertex resulting from the
collapse is, for any dart matched application $4$'s node $n_{4}$, is the face
orbit incident to $n_{4}.@1.@2.\orbit{0,1}$ being the $\orbit{0,1}$-incident to
$n_{6}$ being, this time, the top face expected to be collapsed
(Fig.~\ref{fig:5-eval}). %
