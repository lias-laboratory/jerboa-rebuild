\section{History Tree}

Following the construction of a Parametric Specification with PIs and PNs, the
events that led to the registered topological parameters must be arranged into a
\emph{History Tree} (or \emph{HT}). %
An HT must be issued for each PI. %
It is built in a reverse order, \textit{i.e.} from the last record of a PI up to
the first. %

\subsection{Event nodes and Orbit nodes}

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.25\textwidth]{hr-rejeu-ajout-0-export}
  \caption{History Tree of the Persistent Name $PN_{1}$}
  \label{fig:history-tree-pn1}
\end{figure}

In order to represent the history of an orbit, a HT must arrange more
than just a list of the rules involved and the events they produced. %
Therefore, and with the intent to not miss important data, they comprise two
types of nodes~: \emph{event nodes} and \emph{orbit nodes}. %
Such an arrangement traces an topological parameter back to former orbits it
derives from until these former orbits were created \textit{ex nihilo}. %

An event node (Fig.~\ref{fig:history-tree-pn1}) is a node which represents a
specific event produced by a rule application. %
Therefore, it contains an event type (\textit{i.e.} Creation, Merge, Split and
so on) and is associated to an application number. %

An orbit node (Fig.~\ref{fig:history-tree-pn1}) is a node which represent a
specific orbit resulting from the application of rule given a node of the rule
right member. %
Therefore, it contains an orbit type and is associated to a node. %

Intuitively, each rule application generates a depth level within a HT and each
level is made up of two layers. %
A layer for the event nodes (\textit{i.e.} an \emph{Event Level}), and a layer
for the orbit nodes (\textit{i.e.} an \emph{Orbit Level}). %
For example, the Persistent Name $PN_{1}$ (Fig.~\ref{fig:history-tree-pn1})
designates a face orbit ($\orbit{\alpha 0, \alpha 1}$)
(Fig.~\ref{fig:house-ground}) in order to extrude it. %
In the related HT (Fig.~\ref{fig:history-tree-pn1}), the upper layer is the
Event Level and the lower layer is the Orbit Level. %
The Event Level holds an application number $1$ (in a grey box with rounded
corners) as well as an event node (also with rounded corners). %
The Orbit Level holds a node name, $n_{3}$ (in a blue box with sharp corners) as
well as an orbit node. %
With such an arrangement, a node can only be linked with one the other type,
\textit{i.e.} an event node with an orbit node. %
In order to avoid ambiguity in a HT, an event node can lead to only one orbit
node while an orbit node can lead to one or more event nodes. %


\subsection{Trace and origin}

\begin{figure}[ht!]
  \centering
  \includegraphics[width=0.4\textwidth]{ht-5-top-half}
  \caption{Top half of the $PN_{6}$ History Tree}
  \label{fig:history-tree-pn6}
\end{figure}

In an intent to accurately describe the history of an orbit, a HT defines two
types of relationships between nodes, namely a \emph{trace} (black arrows) and
an \emph{origin} (red arrows). %

A trace relationship indicates that an orbit existed prior to a rule
application. %
Intuitively, these orbits have the same type. %
{\color{darkgreen}%
  For example, the $PN_{6}$ in the parametric specification
  (Table~\ref{tbl:param-spec-example}) designates a vertex
  ($\orbit{\alpha 1, \alpha 2}$) to chamfer
  (Fig.~\ref{fig:house-reshape-flat-to-sharp-roof}). %
  In an excerpt of the related HT (Fig.~\ref{fig:history-tree-pn6}), the face
  vertex orbit ($\orbit{\alpha 1}$), associated to the node $n_{0}$ existed
  prior to the third rule application and, thus, are linked with trace
  relationships. %
}

An origin relationship represents an additional orbit, it is issued when there
is a creation, a split or a merging event. %
For example, in a creation, an origin is the orbit used to copy create another
orbit. in a split or a merging event, it helps to get rid of ambiguities and
offer a precise history description of an orbit. %

\begin{definition}[Origin]\label{def:origin}
  Let $r : L \longrightarrow R$ be a rule, $m : L \to G$ a match,
  % $t : G \longrightarrow^{r,m} H$ a transformation of $G$ by $(r,m)$,
  $\orbit{o}$ an orbit type, $n$ a preserved node of $r$ and $d$ a preserved
  dart of $t$.

  An origin orbit of $R\orbit{o}(n)$ in $L$ is an orbit $L\orbit{o'}(n')$ where:
  \begin{itemize}
    \item $n'$ is a node of $L$
    \item $\orbit{o'}$ is the set of dimensions $i$ where $i$ is rewritten in an
          implicit link of $n'$; \textit{i.e.}
          $(l_{L}(n') \mapsto l_{R}(n))(i) = j \in \orbit{o}$
  \end{itemize}
  An origin orbit of $H\orbit{o}(d')$, where $d'$ is a dart matched by $n'$, is
  an orbit $G\orbit{o'}(d)$.
\end{definition}

When an orbit is the result of a creation event, it cannot be traced. %
In this case, its origin is the orbit that led to its creation. %
{\color{darkgreen}%
  For example, the vertex orbit $R\orbit{\alpha 1, \alpha 2}(n_{2})$ in the face
  triangulation rule (Fig.~\ref{fig:rule-face-triangulation}) is a created
  orbit. %
  Following the definition~\ref{def:origin} and given that
  $\orbit{o} = \orbit{\alpha 1, \alpha 2}$, $R(n_{2})$ first implicit link belongs
  to $\orbit{o}$, therefore, $L(n_{0})$ first implicit link belongs to
  $\orbit{o'}$, such that $\alpha 0 \in \orbit{o'}$. %
  Since the second implicit link of the orbit also belongs to $\orbit{o}$, then,
  $\alpha 1 \in \orbit{o'}$ and $\orbit{o'} = \orbit{\alpha 0, \alpha 1}$. %
  Therefore, the vertex created by the face triangulation rule originates from
  the face being triangulated. %
  %
  In the $PN_{6}$ HT excerpt, the face vertex orbit $\orbit{\alpha 1}$
  associated to the node $n_{6}$ is created by the second rule application and
  its origin is the face vertex orbit associated to the node $n_{5}$ (which is
  created \textit{ex nihilo} since the first rule has no topological
  parameter). %
}%
When an orbit is the result of a split or a merging event, an additional origin
orbit helps to have a more precise history as well as to get rid of ambiguities
at reevaluation time. %
{\color{darkgreen}%
  For example, the face orbit $R\orbit{\alpha 0, \alpha 1}(n_{0})$ in the face
  triangulation rule is a split orbit. %
  Given that $\orbit{o} = \orbit{\alpha 0, \alpha 1}$, $R(n_{0})$ first implicit
  link belongs to $\orbit{o}$, therefore, $L(n_{0})$ first implicit link belongs
  to $\orbit{o'}$, such that $\alpha 0 \in \orbit{o'}$. %
  $R(n_{1})$ has no implicit link within $\orbit{o}$ and $R(n_{2})$ also has
  only the first implicit link within $\orbit{o}$. %
  Therefore, the face orbit $R\orbit{\alpha 0, \alpha 1}(n_{0})$ originates from
  the face edge orbit $L\orbit{\alpha 0}(n_{0})$. In the $PN_{6}$ HT excerpt,
  the face edge orbit ($\orbit{\alpha 0}$) associated to the node $n_{0}$ is
  split by the third rule application. %
  Thus, in order to know from which side of the split the orbit evolved, the
  origin relationship designates a dart (which the larger sub-orbit of
  $\orbit{0}$, excluding $\orbit{0}$ itself). %
}%


\subsection{Relative path to an origin}



\subsection{Example}
A history tree is built from the last known state of an entity, which will be
the root, up to the first entities it originates from, the leaves. %

The events, given the orbit type of an entity, can be deduced from a rule and a
tree is created for each persistent ID of a persistent name. %
Such a tree is built from the last known state of an entity, which will be the
tree root, up to the first entities it originates from, \emph{i.e.} the tree
leaves. %
