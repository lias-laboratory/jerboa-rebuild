During an object's construction, a Parametric Specification records the rules
applied and their parameters (both topological and geometric) in order to
describe the modelling process. %
When a Parametric Specification is edited, rules may be added, deleted, moved,
and their parameters can be changed.%
With such changes, the topological parameters may have different concrete names,
be deleted and so on. %
As a result, using the parameters' concrete names eventually lead to unexpected
results (at best) or failure at runtime. %

To this end, persistent names are required to robustly identify topological
parameters across an object reevaluation. %

This section will follow evaluation (Fig.~\ref{fig:evaluation}) of a modelling
process as an example in order to illustrate the creation of a parametric
specification and its persistent names. %
Its re-evaluation (Fig.~\ref{fig:reevaluation}) illustrates a variant modelling
process where a rule is added between applications 1 and 2, and where the
application 3 is deleted. %


\subsection{Persistent name}

With rules using darts as topological parameters, it ensues that these are the
entities persistent names must robustly identify. %

A dart is created through a unique sequence of rules' applications and their
matching nodes. %

For example, following the initial evaluation, dart $10$'s
(Fig.~\ref{fig:7-eval}) persistent name is the $PN_{7}$. %
However, due to an editing of the parametric specification, this same dart is
given a different concrete name $c$ (Fig.~\ref{fig:7-reeval}) at reevaluation. %
At evaluation, $square$'s (application 1) rule node $n_{3}$ creates the dart $3$
''ex nihilo``. %
Then, $extrusion$'s (application 2) rule node $n_{4}$ copy creates dart $10$
from dart $3$. %
And, finally, dart $10$ is matched by $triangulation$'s (application 5) rule
node $n_0$. %
At reevaluation, $squares$'s rule node $n_{3}$ recreates the dart $3$
identically. %
Now, the added $insertion$'s rule node $n_{0}$ matches the dart $3$ and, there
is a shift in the concrete names attributions from now on. %
Then, $extrusion$'s rule node $n_{4}$ copy creates the dart $12$ from the dart
$3$. %
Finally, dart $12$ is matched by $extrusion$'s rule node $n_{0}$. %
The persistent name built from dart $10$ is $PN_{7}=[1n_{3};2n_{4};5n_{0}]$ and
is able to find that dart $c$'s concrete name is $12$. %

A persistent name being a record of $\PN{application, rule\_node}$ pairs
does robustly identify a dart. %

By construction, a persistent name identifies a unique dart. %
That is, darts all have a different persistent name. %

\subsection{Parametric Specification syntax}


A number of fields are used to describe an application within a parametric
specification, namely an application number \textit{AppNumber}, a \textit{Rule},
the topological parameters \textit{TopoParameters} and the geometric parameters
\textit{Geometric Parameters}. %
%
\begin{verbatim}
  1-square(pos)
  2-extrude(PN1=[1n2], vec)
  3-insert(PN2=[1n5;2n4], vec)
  4-extrude(PN3=[1n5;2n4;3n0], vec)
  5-triangulate(PN4=[1n3;2n2])
  6-collapse(PN5=[1n4;2n6;4n6])
  7-chamfer(PN6=[1n5;2n5;3n0;4n4;6n2])
  8-colour(PN7=[1n3;2n4;5n0])
\end{verbatim}
%
%
The parametric specification above represents the modelling process of an
initial evaluation (Fig.~\ref{fig:evaluation}). %

Additionnaly, a set of tags are used to describe if an application is either
added (\textit{ADD}), deleted (\textit{DELETE}) or moved (\textit{MOVE}) at
reevaluation. %


\begin{verbatim}
   1-square(pos)
   ADD1-insert(3,vec)
   2-extrude(PN1=[1n3], vec)
   DELETE 3-insert(PN2=[1n5;2n4], vec)
   ...
   8-colour(PN7=[1n3;2n3;5n0])
\end{verbatim}
