\section{Parametric specifications}

When reevaluating an object, it is necessary to firstly record its constructive
process, or \emph{parametric specification}. %
Each entry of the parametric specification comprises an application index, the
rule associated to the application index and the topological parameters used by
the rule. %

\subsection{Persistent Identifiers}

Upon the application of a rule, each node matches an orbit and, subsequently,
the darts it is comprising. %
Since these darts can be part of several orbits, it becomes necessary to
persistently identify each one of them when they are matched by a node. %
To allow such an identification, a \emph{persistent ID} needs to be created and
attached to each dart within a G-map. %
A persistent ID, for all the rules that matched a given dart, an application
index and the name of the node that matched this dart. %
Depending on the matched dart, a persistent ID is either created or derived. %
It is created when the dart is created ``ex nihilo'' and updated when the dart
is copy created or preserved. %
When the dart is copy created, the persistent ID is updated by adding from the
one of a dart matched by a neighbouring node (ideally closer to the hook). %
When the dart is preserved, the persistent ID is updated a adding an application
index and a node name. %

{\color{darkgreen} See Fig.~\ref{fig:demo-fig-triangulation-left-match}, dart
  $a_{0}$ persistent ID is $1 — n_{6}$ because it has been created without copy
  by the node $n_{6}$ of the first rule application. %
  However it has been updated after the application of the face triangulation
  rule Fig.~\ref{fig:rule-face-triangulation}. Its new persistent ID is now
  $1 — n_{6}; 2 – n_{0}$ as it has been matched by the node $n_{0}$ of the
  second rule application.} %

\subsection{Persistent Names}

Although darts have persistent IDs, a topological entity comprises several. %
A single persistent ID may not suffice to represent all the rules that impacted
such an entity. %
The gathering of enough persistent IDs within a persistent name can then be
necessary to persistently and formally designate a topological entity. %
A persistent name is created a posteriori of an application. %
It is a collection of one or more persistent IDs found in an entity. %
It must comprises a minimum of persistent IDs and these persistent IDs must
represent a maximum of rule application with as little overlap as possible. %
% On souhaite représenter une entité topo avec un maximum d'information
% d'évolution tout en ne prenant que le nécessaire en PIs.
