During an object's construction, a parametric specification records the rules
applied and their parameters (both topological and geometric) in order to
describe the modelling process. %
When a parametric specification is edited, rules may be added, deleted, moved,
and their parameters can be changed.%
With such changes, the topological parameters may have different concrete names,
be deleted and so on. %
As a result, using the parameters' concrete names eventually lead to unexpected
results (at best) or failure at runtime. %

To this end, persistent names are required to robustly identify topological
parameters across an object re-evaluation. %

This section will follow evaluation (Fig.~\ref{fig:evaluation}) of a modelling
process as an example in order to illustrate the creation of a parametric
specification and its persistent names. %
Its re-evaluation (Fig.~\ref{fig:reevaluation}) illustrates a variant modelling
process where a rule is added between applications 1 and 2, and where the
application 3 is deleted. %


\subsection{Persistent name}

Since rules use darts as topological parameters, it follows that each persistent
name must represent a unique dart's history so that it can be matched at
re-evaluation. %
Well, rules allow the identification of both matched and created darts and, more
specifically, which node matches or create each dart. %

\begin{figure}[ht]
  \centering
  \includegraphics[height=.25\textwidth]{rule-create-square-face}
  \caption{Square creation rule}
  \label{fig:rule-create-square}
\end{figure}

For example, the square rule (Fig.~\ref{fig:rule-create-square}), creates $8$
darts ``ex nihilo'' (Fig.~\ref{fig:1-eval}) one per node. %
Thus, dart $3$ is created by first application's node $n_{3}$ and its history is
noted $[1n_{3}]$. %
Similarly, since darts $5$ and $6$ are created by the square rule application
$1$'s nodes $n_{5}$ and $n_{6}$ respectively, their histories are $[1n_{5}]$ and
$[1n_{6}]$. %

The second application (Fig.~\ref{fig:2-eval} and for more details
Fig.~\ref{fig:before-after-extrusion}), being an extrusion produce by the
extrude rule (Fig.~\ref{fig:rule-face-extrusion}), copy creates darts from the
matched darts. %
Thus, node $n_{1}$ (orange) matches the $8$ darts of the square
(Fig~\ref{1-eval}), hence, nodes $n_{2}$ to $n_{6}$ create darts copied from the
$8$ matched darts. %
As a result, dart $6$'s history is now $[1n_{6};2n_{1}]$ and its copy $8$'s
history is $[1n_{6};2n_{2}]$, its copy $9$'s history is $[1n_{6};2n_{3}]$, its
copy $10$'s history is $[1n_{6};2n_{4}]$ and so on. %
Similarly, darts $46$ and $47$ histories are $[1n_{5};2n_{5}]$ and
$[1n_{5};2n_{6}]$ respectively since they are copy created by the extrude rule's
  nodes $n_{5}$ and $n_{6}$ from dart $5$ which was created by square rule's node
$n_{5}$. %
For the same reasons, darts $33$ and $35$ histories are $[1n_{3};2n_{2}]$ and
$[1n_{3};2n_{4}]$ respectively. %
It occurs that the cube's 48 darts all have a different history. %

\begin{figure}[ht]
  \centering
  \includegraphics[height=35pt]{rule-insert-vertex}
  \caption{Insert vertex rule}
  \label{fig:rule-insert-vertex}
\end{figure}

Continuing with the third application (Fig.~\ref{fig:3-eval}) being the
insertion of a vertex on an edge with the insert rule
(Fig.~\ref{fig:rule-insert-vertex}). %
Darts $46$ and $47$ are matched by the insert rule's node $n_{0}$. %
Their histories now are $[1n_{5};2n_{5};3n_{0}]$ and $[1n_{5};2n_{6};3n_{0}]$
respectively. %
Conversely, darts $33$ and $35$ are not matched during the vertex insertion and
their histories remain $[1n_{3};2n_{2}]$ and $[1n_{3};2n_{4}]$ respectively. %
The object's 52 darts, again, all have a different history. %
The same goes for the next steps' darts in this or any other construction made
with rules. %

Darts histories, thus, represent in a unique way any dart, hence, their usage as
persistent names. %
Therefore, $PN_{1}=[1n_{6}]$, $PN_{2}=[1n_{5};2n_{5}]$,
$PN_{3}=[1n_{5};2n_{6};3n_{0}]$ and so on. %

\subsection{parametric specification syntax}


A number of fields are used to describe an application within a parametric
specification, namely an application number \textit{AppNumber}, a \textit{Rule},
the topological parameters \textit{TopoParameters} and the geometric parameters
\textit{Geometric Parameters}. %
%
\begin{figure}[h!]
  \centering
    \begin{verbatim}
      1-square(pos)
      2-extrude(PN1=[1n6], vec)
      3-insert(PN2=[1n5;2n4], vec)
      4-extrude(PN3=[1n5;2n4;3n0], vec)
      5-triangulate(PN4=[1n3;2n2])
      6-collapse(PN5=[1n4;2n6;4n6])
      7-chamfer(PN6=[1n5;2n5;3n0;4n4;6n2])
      8-colour(PN7=[1n3;2n4;5n0])
    \end{verbatim}
  \caption{init-spec}
  \label{fig:init-spec}
\end{figure}
%
%
The parametric specification above represents the modelling process of an
initial evaluation (Fig.~\ref{fig:evaluation}). %

Furthermore, a set of tags are used to describe if an application is either
added (\textit{ADD}), deleted (\textit{DELETE}) or moved (\textit{MOVE}) at
reevaluation. %


\begin{verbatim}
   1-square(pos)
   ADD1-insert(3,vec)
   2-extrude(PN1=[1n3], vec)
   DELETE 3-insert(PN2=[1n5;2n4], vec)
   ...
   8-colour(PN7=[1n3;2n3;5n0])
\end{verbatim}
