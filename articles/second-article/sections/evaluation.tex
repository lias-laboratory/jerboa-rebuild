\section{Evaluation}

Although a persistent name represents the history of a dart, an orbit is subject
to topological changes and, thus, requires the construction of its own history
in order to be accurately matched at re-evaluation. %
Once the initial evaluation (Fig.~\ref{fig:evaluation}) is done and its
parametric specification (Fig.~\ref{fig:init-spec}) is built, an evaluation's Directed
Acyclic Graph (or DAG) must be issued for each persistent name before
any parametric specification can be re-evaluated. %
An evaluation DAG traces the history of each topological parameter back to the
first created orbits it originates from, thus allowing the matching of the
corresponding topological parameter at re-evaluation time. %

\subsection{Evaluation DAG}

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.25\textwidth]{pn3}
  \caption{$PN_{3}$'s evaluation DAG}
  \label{fig:eval-dag-pn3}
\end{figure}

An evaluation DAG is built parsing %with
the applications and nodes
of a persistent name from end to start. %
It is sorted by levels representing the different applications
inside a history. %
Each level is made of an orbit level and an event level.
The orbit level contains a node's name and some orbits.
The event level contains a rule's application number and some events. %

For example, let us consider $PN_{3}=[1n_5;2n_6;3n_0]$
(Fig.~\ref{fig:4-eval}). %
$PN_3$ represents the topological parameter upon which the face extrusion rule
is applied. %
This rule is filtered by hook $n_1$, matching the face $\orbit{0, 1}(47)$
(Fig.~\ref{fig:3-eval}). %
The matched orbit's history is built from its dart's history (\textit{i.e.} its
persistent name). %
The DAG is built bottom-up by a backward traversal through the persistent
name. %
Since $PN_3$ is made of 3 parts, its evaluation DAG contains 3 levels. %
The last part of $PN_3$ is $3n_0$, meaning that the dart of interest is filtered
by the node $n_0$ of the third operation in the initial parametric
specification. %
Therefore, the Orbit level 3 contains both $n_0$ and the matched orbit
$\orbit{0, 1}$. %
As shown in (Fig.~\ref{fig:init-spec}), the third operation is the vertex
insertion on an edge (Fig.~\ref{fig:rule-insert-vertex}). %
The right side of this rule has modified the orbit $\orbit{0,1}(n_0)$. %
We infer that the Event level 3 contains the third operation (\verb+3-insert+)
and the event's name \texttt{MODIFICATION}. %

Continuing through the persistent name's backward traversal, the previous
element \texttt{2n6} allows the DAG's second level computing in a similar way. %
Finally, \texttt{1n5} allows computing the first level. %

At last, the produced evaluation DAG (Fig~\ref{fig:eval-dag-pn3}) represents the
volume face $\orbit{0,1,3}$-orbit's history being resulting from applying
\texttt{4-extrusion}. %
This DAG can be read top-down:
\begin{description}
    \item[Level 1] The application \texttt{1-square} creates the volume face $\orbit{0,1}(n_5)$.
    \item[Level 2] From this volume face, \texttt{2-extrusion} creates the volume face $\orbit{0,1}(n_6)$.
    \item[Level 3] Finally, this latter volume face is modified by \texttt{3-insert} inserting a vertex on its edge.
\end{description}

The extrusion rule's application matches face $\orbit{0,1}(n_{5})$. %
Then the extrusion creates the face $\orbit{0,1}(n_{6})$
(Fig.~\ref{fig:2-eval}). %
Finally, the vertex insertion rule's application modifies the face
$\orbit{0,1}(n_{0})$ (Fig.~\ref{fig:3-eval}). %

The syntactic analysis of the rules enables events to be computed only once. %
These events can be stored in cache to automatically build other evaluation
DAGs. %

\subsection{Traces and origins}

\begin{figure}[hbt]
  \centering
  \includegraphics[width=0.3\textwidth]{pn7}
  \caption{$PN_{7}$'s evaluation DAG}
  \label{fig:eval-dag-pn7}
\end{figure}

In order to accurately represent the history of an orbit, two types of arrows
are used in an evaluation DAG (and later in the re-evaluation DAG): black trace
arrows and red origin arrows. %

A \emph{black trace arrow} allows orbit evolution tracing. %
For example, the triangulation rule splits an initial face into multiple
subfaces. %
If one of these subfaces is referenced in the DAG, it is connected by a trace
arrow to the initial face. %
Therefore, a trace arrow connects two orbits of the same dimension. %
A \emph{red origin arrow} allows linking an orbit with the orbit that generated
it, thereby connecting two orbits of potentially different dimensions. %
For example, continuing with the triangulation rule
(Fig.~\ref{fig:rule-triangulation}), we observe that upon applying this rule,
each edge of the initial face generates a different subface. %
If any of these subfaces is referenced in the DAG, it is then connected by an
origin arrow to the edge that generated it. %
This is what can be observed considering again the example in
Fig.~\ref{fig:evaluation} and more precisely the colour rule's $PN_7$
parameter. %
$PN_7$ represents dart $35$'s history and $PN_7$'s evaluation DAG
(Fig.~\ref{fig:eval-dag-pn7}) represents the history of the volume face that
needs to be colored (the volume face adjacent to dart $35$). %
This volume face $\orbit{0,1}(n_0)$ is the result of the 5-triangulation rule
which splits the volume face $\orbit{0,1}(n_4)$ and has the face edge
$\orbit{0}(n_4)$ as its origin (respectively represented by a black and red
arrow between orbit level 2 and event level 3). %

As explained in section 4.1, this DAG is built using a  traversal of $PN_7$ and a bottom-up construction. This process is done in a similar way for both traces and origins, allowing for an efficient persistent naming mechanism that also takes into account the impact of origin modifications during reevaluation.
To illustrate this, let us consider the previous face edge $\orbit{0}(n_4)$, which is the origin of the volume face $\orbit{0,1}(n_0)$ that needs to be colored. Suppose that, due to the addition of an operation in the edited specification, this origin is eventually split into two face edges. Upon applying the 5-triangulation, these two face edges will generate two volume faces, which can then be matched during reevaluation to the face to be colored, leveraging on the  origin orbit recorded in the DAG.

In a formal way, an origin orbit can be automatically deduced through the syntactic analysis of a rule.
More precisely, if $n$ is a hook and $n'$ is not a preserved node different from $n$, the origin of an orbit $R\orbit{o}(n')$ is the sub-orbit $L\orbit{o'}(n)$ consisting of the set of $n$'s implicit arcs that are:
\begin{itemize}
    \item rewritten on $R\orbit{o}(n')$;
    \item not rewritten on $R\orbit{o}(n')$ and belonging to $\orbit{o}$.
\end{itemize}

For example, let us calculate the origin of a volume face ($\orbit{0,1}$-orbit) resulting from a split in the triangulation rule (Fig.~\ref{fig:rule-triangulation}). Only the implicit arc $0$ in the $L\orbit{0,1}$-orbit incident to the hook $n_0$ is rewritten on $R\orbit{0,1}(n_0)$). Hence, the origin orbit of a volume face split by the triangulation rule is a face edge ($\orbit{0}$-orbit).

When the orbit is either \emph{split} or \emph{merged}, the syntactic analysis
allows the deduction of an \emph{origin} in addition to a traced orbit. When the
orbit is \emph{created}, if the left-hand side of the rule is empty (meaning the
orbit is created from scratch), there is neither trace nor origin (event level 1
in Fig.~\ref{fig:eval-dag-pn7} where the 1-square rule creates the edge face
$\orbit{0}(n_3)$ from scratch). Otherwise, there is no trace but an origin
(event level 2 in Fig.~\ref{fig:eval-dag-pn7} where the 2-extrusion rule creates
the front face $\orbit{0,1}(n_4)$ of the cube from the origin previous face edge
$\orbit{0}(n_3)$). When the orbit is just \emph{modified} or \emph{not
  modified}, there is only a traced orbit (event level 3 in
Fig.~\ref{fig:eval-dag-pn3} where the 3-insert rule modifies the top face
$\orbit{0,1}(n_6)$ of the cube inserting a vertex on its boundary).


\subsection{Paths to origins}

\begin{figure}[hbt]
  \centering
  \includegraphics[height=35pt]{rule-face-collapse}
  \caption{Face collapse rule}
  \label{fig:rule-face-collapse}
\end{figure}

We have defined the origin of an orbit $R\orbit{o}(n')$ when $n'$ is not a
preserved node different from the hook. In the opposite case, it is necessary to
add to the origin the \emph{path} that allows reaching the implicit arcs of the
hook from $n'$, because the implicit arcs of an origin are those of the hook,
not those of $n'$.

For example, in the face collapse rule (Fig.~\ref{fig:rule-face-collapse}), node $n_2$ is a preserved node different from the hook. Assume we want to define the origin of the volume vertex $\orbit{1,2}(n_2)$.
A path represents the traversal in the right-hand side of the rule from the node $n_{2}$ to the hook node $n_{0}$. The traversed explicit arcs from $n_{2}$ to $n_{0}$ are, in the following order, $1$ and $2$ (written $@1.@2$ in the evaluation DAG).
This can be seen in the $PN_{6}$'s evaluation DAG (Fig.~\ref{fig:eval-dag-pn6}), where the application of the 6-collapse rule generates a merge of vertices and the origin of the volume vertex $\orbit{1,2}(n_{2})$ is the volume face $@1.@2.\orbit{0,1}(n_{4})$.
Concretely, node $n_4$ of the extrusion rule (Fig.~\ref{fig:rule-face-extrusion}) used at the previous level of the DAG (level 4) matches dart $55$ on the lateral faces of the cube (Fig.~\ref{fig:before-after-extrusion}) because it is the dart matching the history stored in $PN_6$ DAG (\verb+1-square+ creates the initial bottom volume face $\orbit{0,1}(n_5)$. Then, \verb+2-extrusion+ applied on this bottom face creates the volume face $\orbit{0,1}(n_6)$ which is then modified by \verb+3-insert+ in $\orbit{0,1}(n_0)$. Finally, \verb+4-extrusion+ applied on this modified volume face creates node $n_4$ which matched dart $55$).
The volume face $@1.@2.\orbit{0,1}(n_{4})$ reached starting from dart 55 and following links 1 and 2 is indeed the top face expected to be collapsed.
